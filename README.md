**Project Overview**

The purpose of this project is to build a small HTTP server in pure Python using sockets in order to understand how web communication works at a low level. Instead of relying on web frameworks such as Flask or Django to handle requests and responses automatically, this project focuses on manually implementing the core components of an HTTP server.

The project is not primarily about building a complex application. Its main purpose is to understand the HTTP protocol and the request–response model by implementing the server from first principles. By constructing the server manually, the internal behavior of web frameworks will become much clearer later.

---

**Purpose of the Project**

This project serves as a bridge between theoretical knowledge of HTTP and practical backend engineering. Concepts such as request lines, headers, message bodies, status codes, content types, routing, JSON responses, cookies, and redirects will be implemented directly rather than hidden behind a framework.

By completing this project, the process of web communication will become concrete rather than abstract. Later, when using frameworks such as Flask, it will be clear what the framework is doing internally and why those abstractions exist.

---

**Main Goal of the Project**

The main goal is to build a working HTTP server in Python that can:

- listen on a local port
- accept browser requests
- read raw HTTP messages
- parse HTTP request data
- route requests based on the path
- return valid HTTP responses
- support multiple routes
- return HTML, plain text, and JSON
- handle unknown routes correctly using a 404 response

This server will function as a basic request–response system.

---

**High-Level System Behavior**

At a conceptual level, the server will operate according to the following flow:

Browser or Client

→ sends HTTP request

→ Python HTTP server receives request

→ server parses the request

→ server decides how to handle the request

→ server builds an HTTP response

→ server sends the response back

→ browser displays the result

This system can be understood as a **request–response machine**.

---

**Core Server Responsibilities**

The server must perform the following technical tasks:

- open a socket
- bind to a host and port
- listen for incoming connections
- accept a client connection
- receive request bytes
- decode the request into text
- parse the HTTP request
- route the request to the correct handler
- build an HTTP response
- send response bytes back to the client
- close the connection

---

**HTTP Responsibilities of the Server**

In addition to networking behavior, the server must understand HTTP concepts such as:

- HTTP method
- request path
- HTTP version
- request headers
- message body (later stage)
- status codes
- response headers
- content types

This ensures that the server communicates using valid HTTP messages.

---

**Supported Routes**

The server will initially support the following routes:

- `/`
- `/about`
- `/api/hello`

Any unknown route should return a **404 Not Found** response.

---

**Response Formats**

The server must be able to return different content types, including:

- HTML
- plain text
- JSON

This ensures that the server can behave both like a web server and like an API server.

---

**Definition of the Minimum Viable Product (MVP)**

The Minimum Viable Product is the smallest version of the server that demonstrates a full HTTP request–response cycle.

The MVP server must:

- start on `127.0.0.1:8080`
- accept browser requests
- parse the request line and headers
- route requests based on the path
- send valid HTTP responses
- include required response headers
- return HTML, plain text, and JSON responses
- return 404 responses for unknown routes

If all of these features work correctly, the MVP is complete.

---

**Required HTTP Response Structure**

Every HTTP response generated by the server must contain:

1. Status line
2. Response headers
3. Blank line
4. Message body

At minimum, responses must include the following headers:

- `Content-Type`
- `Content-Length`

This ensures that browsers can correctly interpret the response.

---

**Implementation Milestones**

The project should be implemented in stages, each adding a new capability.

**Milestone 1 — Minimal HTTP Response**

Build a socket server that accepts a connection and sends a hardcoded HTTP response. Opening the server in a browser should display a simple message.

**Milestone 2 — Read Raw HTTP Request**

Modify the server so that it receives the browser’s HTTP request and prints the raw request text in the terminal.

**Milestone 3 — Parse the Request Line**

Extract and display the HTTP method, path, and version from the request line.

**Milestone 4 — Parse Headers**

Parse the request headers into a dictionary structure so individual headers can be accessed.

**Milestone 5 — Implement Routing**

Add routing logic so different paths return different responses.

**Milestone 6 — Build a Response Generator**

Create a structured way to generate HTTP responses including status codes, headers, and body.

**Milestone 7 — Add 404 Handling**

Return a proper 404 response when a route does not exist.

**Suggested Project File Structure**

Initially, the project should remain simple:

```
http_server_project/
└── server.py
```

After the MVP is complete, the project can be refactored into multiple modules:

```
http_server_project/
├── server.py
├── request.py
├── response.py
├── router.py
├── handlers.py
└── utils.py
```

However, early over-engineering should be avoided. The first goal is to make the server work.

---

**Project Constraints**

To ensure the project remains educational and focused:

- Use pure Python only
- Use the `socket` module instead of web frameworks
- Avoid libraries that hide HTTP behavior
- Keep the code explicit and readable
- Prioritize understanding over optimization
- Implement features step by step

---

**Possible Extensions After the MVP**

After the basic server works, additional features can be implemented, such as:

- Support for POST requests
- Request body parsing
- JSON request handling
- Redirect responses
- Cookies
- Simple sessions
- Static file serving
- File uploads
- Query parameter parsing
- Logging and caching headers

These features will gradually transform the basic server into a more complete HTTP server.

---

**Final Project Objective**

The ultimate purpose of this project is to deeply understand how HTTP communication works by building an HTTP server from scratch. By the end of the project, it should be possible to clearly explain how a browser communicates with a server, how HTTP requests and responses are structured, how routing works, how content types are handled, and how web frameworks simplify these processes.
